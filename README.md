# Switchboard — a URL shortener with analytics

A production-shaped URL shortener: short-code generation, click analytics
(device/browser/referrer), Redis caching with graceful degradation, JWT
auth, rate limiting, and a small dashboard. Built to demonstrate backend
system-design thinking, not just CRUD.

## Why this project

Anyone can build a URL shortener in an afternoon. What makes it interview-
worthy is the set of decisions underneath it — how do you generate codes
without collisions, how do you keep redirects fast at scale, what happens
when your cache goes down. This README documents those decisions on
purpose; that's the part worth talking about.

## Quick start

```bash
git clone <your-repo-url>
cd url-shortener
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

# Optional: copy .env.example to .env and edit. Works with zero config too —
# defaults to a local SQLite file with no Redis required.
cp .env.example .env

uvicorn app.main:app --reload
```

Then open:
- **http://localhost:8000/dashboard/** — the UI
- **http://localhost:8000/docs** — interactive Swagger API docs (auto-generated by FastAPI)

### Run with Docker (API + Redis + Postgres)

```bash
docker compose up --build
```

### Run tests

```bash
pytest
```

## Architecture

```
Browser/client
     │
     ▼
FastAPI app (app/main.py)
     │
     ├── /auth/*        register, login (JWT)
     ├── /links/*        create, list, get, delete, analytics
     ├── /{short_code}   redirect (the hot path)
     │        │
     │        ├──► Redis cache (checked first)
     │        └──► Postgres/SQLite (source of truth, on cache miss)
     │
     └── /dashboard      static HTML/JS UI
```

## Key design decisions

**Short-code generation: base62(auto-increment ID), not random strings.**
Random codes need a collision-check-and-retry loop against the database on
every single creation. Encoding the row's own primary key in base62
guarantees uniqueness for free — no retry loop, no collision handling code
at all. The trade-off (worth raising in an interview): codes are loosely
ordered/guessable relative to each other since they derive from a sequential
ID. A portfolio project should take the simpler, defensible trade — see
`app/utils.py` for the full writeup.

**Cache-aside with graceful degradation, not a hard Redis dependency.**
Every redirect checks Redis before touching the database, because redirects
are by far the highest-traffic endpoint. But if Redis is down or was never
configured, `app/cache.py` catches the connection error and falls back to
querying the database directly — the app never crashes because a cache was
unreachable. This is a genuine production pattern: caches should degrade,
never become a single point of failure.

**Anonymous link creation, optional ownership.**
`POST /links` doesn't require auth — you can shorten a link anonymously,
same as bit.ly. But if you're logged in, the link auto-associates with your
account (`get_current_user_optional` in `app/security.py`), which is what
enables "my links" listing and lets you later deactivate them.

**Click logging happens synchronously (with a documented path to async).**
Right now each redirect writes a `Click` row before returning the redirect.
For a resume project this keeps the code simple and testable. The natural
next step — and a good "what would you improve" answer — is pushing click
logging onto a background queue (Celery/RQ) so the redirect response
doesn't wait on a database write at all.

**Rate limiting on link creation, not on redirects.**
`POST /links` is rate-limited (`slowapi`) because it's the endpoint someone
could abuse to spam-generate links. Redirects aren't rate-limited because
that's the entire point of the service — every real click needs to work.

## API overview

| Method | Path | Auth | Description |
|---|---|---|---|
| POST | `/auth/register` | — | Create an account |
| POST | `/auth/login` | — | Get a JWT access token |
| POST | `/links` | optional | Create a short link (custom alias, expiry supported) |
| GET | `/links` | required | List your links |
| GET | `/links/{code}` | — | Get link info |
| DELETE | `/links/{code}` | required (owner) | Deactivate a link |
| GET | `/links/{code}/analytics` | — | Click totals, time series, referrer/device/browser breakdown |
| GET | `/{code}` | — | Redirect to the long URL (logs a click) |

Full interactive docs (with try-it-out) are at `/docs` once the server is running.

### Example: create and use a short link

```bash
curl -X POST http://localhost:8000/links \
  -H "Content-Type: application/json" \
  -d '{"long_url": "https://example.com/a/very/long/path"}'
# → { "short_code": "3", "short_url": "http://localhost:8000/3", ... }

curl -i http://localhost:8000/3
# → 302 redirect to https://example.com/a/very/long/path
```

## What I'd add next (good talking points for interviews)

- Move click logging to a background queue so redirects don't wait on a DB write
- Alembic migrations instead of `create_all()`
- Real IP-based geolocation for the analytics dashboard
- Bulk link creation / CSV export
- Per-user rate-limit tiers instead of a flat global limit

## Resume bullet ideas

- Built a URL-shortening service (FastAPI, PostgreSQL, Redis) with JWT auth,
  click analytics, and cache-aside redirect resolution
- Designed a collision-free short-code scheme using base62-encoded
  auto-increment IDs, eliminating retry logic needed by random-code approaches
- Implemented graceful cache degradation so the service stays available if
  Redis becomes unreachable
- Wrote a pytest suite covering auth, link lifecycle, expiry, and analytics
  with an isolated in-memory test database
